1. Estructura de Directorios del Proyecto

Antes de inicializar Git, es un requisito estricto ordenar el espacio de trabajo siguiendo el patrón estándar del programa de formación:

Estructura del Proyecto 
taller3/
├── app/
│   ├──  forms.html
│   ├──  table.html
│   └──  github.html
├── assets/
│   ├── css/ // Hojas de estilo locales
│   ├── img/ // Recursos gráficos e imágenes
│   └── js/  // Scripts funcionales
├──  index.html
└──  README.md

2. Configuración de Identidad Global

Abre tu terminal integrada (Ctrl + Shift + `) y declara tus credenciales institucionales. Esto evita que tus firmas de autor aparezcan anónimas en GitHub.

bash - Terminal 
git config --global user.name "Alejandro Mesa"
git config --global user.email "tu_correo@example.com"

// Comando de verificación:
git config --global --list

3. Ciclo Local (Init, Track & Commit)

Convierte el directorio de tu máquina en un repositorio rastreado por Git y efectúa la primera confirmación de cambios en tu historial local:

Inicializar el repositorio

Crea la base de datos interna oculta llamada .git.

git init
Indexar los archivos (Staging Area)

Prepara los archivos agregándolos al área de preparación de confirmaciones.

git add .
Crear la Confirmación de Estado (Commit)

Guarda una instantánea inmutable en el historial con un mensaje descriptivo.

git commit -m "feat: estructura inicial del taller 3 con bootstrap"

4. Conexión y Subida a GitHub Remoto

Crea un repositorio vacío en tu perfil web de GitHub, luego vincula tu terminal local y sube los cambios directamente a la rama principal:

bash - Cloud Push 
// 1. Asegurar el nombre estándar de la rama principal (main)
git branch -M main

// 2. Enlazar el repositorio local con el de GitHub
git remote add origin https://github.com/tu-usuario/nombre-repositorio.git

// 3. Subir el código por primera vez emparejando la rama
git push -u origin main

5. Flujo de Trabajo Técnico (Ramas / Branches)

En el desarrollo profesional nunca se trabaja directo sobre main. Las nuevas vistas o correcciones de layouts se despliegan en ramas independientes.

Buena práctica: Crea una rama secundaria separada (por ejemplo, para el formulario o para la tabla de datos) para no estropear la versión estable del proyecto.
bash - Ramificaciones 
// Crear una nueva rama dedicada al desarrollo del Formulario
git checkout -b feature-formulario

// Realizas tus modificaciones de código en forms.html, guardas y confirmas:
git add .
git commit -m "style: diseño del formulario optimizado con grids de bootstrap"

// Sincronizar la rama hija de forma independiente con GitHub
git push origin feature-formulario
